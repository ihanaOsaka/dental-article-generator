# MeshiPick サジェストアルゴリズム設計書

## 1. 設計思想

### 目指すもの
> 「あなたの頭の中に浮かんでいる選択肢はコレです」を言い当て、さらにその先を行く提案をする

単なる「最近食べてないものを出す」ではない。**人間の食事選択の心理メカニズム**をモデル化し、ユーザーが「そうそう、それ！」と感じるサジェストと、「あ、それがあったか！」という気づきの両方を提供する。

### 3つの提案枠

| 枠 | 名称 | 役割 | 比率 |
|----|------|------|------|
| 1 | **ドンピシャ枠** | ユーザーの頭の中にあるであろう選択肢 | 50% |
| 2 | **忘れてた枠** | かつて気に入っていたが最近忘れている選択肢 | 30% |
| 3 | **冒険枠** | 新しい発見・未体験の選択肢 | 20% |

## 2. コンテキストレイヤー（フィルタリング）

サジェストの前段階として、**物理的に可能な選択肢**を絞り込む。

### 2.1 時間コンテキスト

```
時間帯判定:
  朝食: 6:00 - 10:00  → 朝食向けの選択肢
  昼食: 11:00 - 14:00 → ランチ営業のお店、手早い料理
  夕食: 17:00 - 21:00 → ディナー営業のお店、じっくり料理
  間食: それ以外      → 軽食・カフェ系

残り時間の推定:
  「今から食べ始めるまでの時間」を逆算
  - 外食（近場500m）: 移動5分 + 待ち時間考慮
  - 外食（中距離2km）: 移動15-20分
  - 外食（遠距離5km+）: 移動30分+
  - 自炊（簡単）: 調理15-30分
  - 自炊（普通）: 調理30-60分
  - 自炊（本格）: 調理60分+
```

### 2.2 位置コンテキスト

```
現在位置からの距離圏:
  圏1: 500m以内（徒歩5分）   → 「すぐ食べたい」時の候補
  圏2: 2km以内（徒歩20分/自転車10分） → 通常の外食候補
  圏3: 5km以内（車/電車15分） → 「少し足を伸ばす」候補
  圏4: 自宅               → 自炊候補

ユーザーの現在位置を判定:
  - 自宅付近 → 自炊の選択肢を増やす
  - 職場付近 → ランチ候補を優先
  - 外出先   → 周辺のお店を優先
```

### 2.3 曜日・状況コンテキスト

```
曜日パターン:
  平日ランチ → 時間制約あり、コスパ重視
  平日ディナー → 疲れ具合考慮、手軽さ重要
  休日ランチ → 時間余裕あり、冒険OK
  休日ディナー → 特別感、外食率UP

天候（将来的に天気API連携）:
  雨天 → 近場優先、デリバリー候補追加
  猛暑 → 冷たいもの、近場優先
  寒い → 温かいもの、鍋系UP
```

## 3. スコアリングモデル

### 3.1 基本スコア = 嗜好スコア × コンテキスト適合度 × 心理モデル補正

各候補（お店 or レシピ）に対してスコアを算出し、ランキングする。

---

### 3.2 嗜好スコア（Preference Score）

**BG/NBDモデル（Beta-Geometric/Negative Binomial Distribution）の応用**

元々は顧客の購買行動モデリングに使われるモデルだが、「ユーザーがお店に行く頻度」のモデリングに最適。

```
各お店/レシピに対して:
  λ (来店/調理頻度): ポアソン過程に従う
  p (離脱確率): そのお店に「飽きた」確率

BG/NBDモデルのパラメータ:
  - frequency: 過去の訪問回数
  - recency: 最後の訪問からの経過日数
  - T: ユーザーの観測期間（アプリ利用開始からの日数）

→ P(Alive | data) = そのお店がまだユーザーの選択肢に「生きている」確率
→ E(X | Alive) = 今後の期待訪問頻度
```

**直感的な意味**:
- よく行くお店（高λ）で最近も行っている（低recency）→ ドンピシャ枠
- よく行っていた（高frequency）が最近行っていない（高recency）→ P(Alive)が中程度 → 忘れてた枠
- P(Alive)が極めて低い → もう飽きた/合わなかった → スコアダウン

### 3.3 心理モデル補正

#### A. 飽き関数（Satiation Function）

同じものを連続で食べると飽きるメカニズムをモデル化。

```
飽き度(item, t) = Σ exp(-α × (t - t_i))
  t_i: 過去にそのアイテムを食べた時刻
  α: 飽き減衰パラメータ（ジャンルにより異なる）

例:
  ラーメン（α=0.3）: 3日前に食べた → 飽き度 0.41 → まだちょっと...
  カレー（α=0.5）: 3日前に食べた → 飽き度 0.22 → そろそろいける
  ごはん/パスタ等の主食（α=1.0）: すぐ飽きが解消 → 毎日でもOK

ジャンル単位でも飽き度を計算:
  中華を3日連続 → 中華ジャンル全体の飽き度UP
```

#### B. 想起確率モデル（Availability Heuristic）

「ユーザーの頭に浮かびやすいか」をモデル化。人間は**最近の経験**と**印象的な経験**を想起しやすい。

```
想起スコア(item) = w1 × recency_score + w2 × frequency_score + w3 × impression_score

  recency_score: 最近食べた/見たものほど高い（指数減衰）
  frequency_score: よく食べるものほど高い（対数スケール）
  impression_score: 高評価をつけたものほど高い

「ドンピシャ枠」: 想起スコアが高い × 飽き度が低い
「忘れてた枠」: 想起スコアが低い × 嗜好スコア(P(Alive))が高い
```

#### C. 多様性欲求モデル（Variety Seeking）

```
人間は一定の多様性を求めるが、その程度は個人差がある。

variety_seeking_index = ユーザーの過去データから推定
  - 毎日違うものを食べる人 → 高い
  - 同じお店にリピートする人 → 低い

多様性補正:
  直近N食のジャンル分布を計算
  偏りが大きい → 違うジャンルにボーナス
  （ただしvariety_seeking_indexが低いユーザーには控えめに）
```

#### D. 意思決定疲れモデル（Decision Fatigue）

```
選択肢が多すぎると人は選べない（ジャムの法則）

サジェスト表示数:
  通常: 3〜5件
  「もっと見る」で追加表示

選択肢の提示順:
  1位: 最もスコアが高い「ドンピシャ」
  2位: 対照的な選択肢（外食↔自炊、和↔洋など）
  3位: 「忘れてた枠」or「冒険枠」
```

## 4. 学習・フィードバックループ

### 4.1 暗黙的フィードバック

```
ユーザーの行動から学習（明示的な入力不要）:

  サジェスト → 採用した    → そのアイテムのスコアUP
  サジェスト → 無視した    → 微減（1回では判断しない）
  サジェスト → 連続無視    → スコアダウン
  自発的に記録 → 記録した  → そのアイテムへの関心を反映

  お店の訪問間隔パターン → λの更新
  ジャンルの偏り変化     → variety_seeking_indexの更新
```

### 4.2 明示的フィードバック

```
  評価（5段階 or いいね） → 嗜好スコアに直接反映
  「もう行かない」       → P(Alive)を0に
  「気分じゃない」       → 一時的なスコアダウン（飽き関数に反映）
```

## 5. 統合スコアリング式

```
最終スコア(item) =
    preference_score(item)           # BG/NBDベースの嗜好
  × context_fitness(item, now)       # 時間・場所・天候の適合度
  × (1 - satiation(item, now))      # 飽き度の逆数
  × variety_bonus(item, recent_meals) # 多様性ボーナス
  × freshness_factor(item)           # 新鮮さ（冒険枠用）
  + user_rating_bonus(item)          # ユーザー評価ボーナス

枠の振り分け:
  ドンピシャ枠: availability_score(item) > threshold_high
               AND satiation(item) < threshold_low

  忘れてた枠:  availability_score(item) < threshold_low
               AND preference_score(item) > threshold_high
               AND P(Alive | data) > 0.3

  冒険枠:     visited_count == 0
              AND context_fitness > threshold_high
              AND (nearby OR similar_to_liked_items)
```

## 6. コールドスタート問題への対処

アプリ利用初期はデータが少ないため、段階的にアルゴリズムを進化させる。

```
Stage 0（登録直後）:
  → Google Mapsのブックマーク・履歴をインポート
  → 位置情報ベースの周辺おすすめ（Google Places評価ベース）
  → 「好きなジャンルを3つ選んでください」で初期嗜好を取得

Stage 1（食事記録 10件〜）:
  → 飽き関数が機能し始める
  → 時間帯・曜日パターンの学習開始

Stage 2（食事記録 30件〜）:
  → BG/NBDモデルのパラメータ推定が安定
  → 想起確率モデルが機能
  → 「忘れてた枠」が精度向上

Stage 3（食事記録 100件〜）:
  → variety_seeking_indexの推定が安定
  → 全アルゴリズムがフル稼働
  → パーソナライズ精度が高い状態
```

## 8. 数理モデルの詳細（参考）

### ポアソン過程とBG/NBDモデル

人間の購買（来店）行動がポアソン分布に従うという知見を活用する。

```
ポアソン分布:
  P(k回訪問 | 期間T) = (λT)^k × e^(-λT) / k!
  λ: 単位時間あたりの来店率（お店ごとに異なる）

ただし、人間は「飽きる」ため、純粋なポアソンでは不十分。
BG/NBDモデルでは:
  - 各取引（来店）後に確率pで「離脱」（飽きる/忘れる）
  - 離脱していない間はポアソン過程で来店

  これにより:
  - 「最近行ってないけどまた行きそう」（生存中だが低λ）
  - 「もう行かなくなった」（離脱済み）
  を区別できる

パラメータ推定:
  r, α (来店頻度のガンマ分布パラメータ)
  a, b (離脱確率のベータ分布パラメータ)
  → ユーザーの全来店データからMLEまたはベイズ推定
```

### 飽き関数の数理

```
指数減衰カーネル:
  K(Δt) = exp(-α × Δt)

飽き度 S(item, t):
  S = Σ_i K(t - t_i)  ※ t_i は過去に食べた各時刻

性質:
  - 直後: S ≈ 1（飽きている）
  - 時間経過: S → 0（飽きが解消）
  - 連続で食べる: S が累積（もっと飽きる）

αの値はジャンルごとに異なる（学習 or 手動設定）:
  高α（飽きにくい）: 主食系（米、パン、パスタ）、定番料理
  低α（飽きやすい）: 特徴的な味（ラーメン、カレー、エスニック）
```

## 9. 将来的な拡張アイデア

- **協調フィルタリング**: 一般公開後、似た嗜好のユーザーの選択を参考に
- **季節性モデル**: 鍋は冬、冷やし中華は夏
- **感情モデル**: 「疲れた日はこってり」「元気な日はヘルシー」
- **社会的コンテキスト**: 一人 vs 複数人、接待 vs カジュアル
- **GPT/Claude連携**: 自然言語で「今日は軽めがいいな」→ コンテキスト反映
